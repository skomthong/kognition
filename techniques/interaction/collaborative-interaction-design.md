# Collaborative Interaction Design

## Summary

Collaborative interaction design creates digital environments where groups of learners work together — co-creating, debating, building shared artifacts, and giving peer feedback. It's the interaction-design complement to [collaborative learning](../../principles/collaborative-learning.md) theory: how do you build the TECHNOLOGY that supports group learning?

## Interaction Patterns

| Pattern | What happens | Tools |
|---|---|---|
| **Shared document editing** | Multiple learners build a single artifact together in real time | Google Docs, Microsoft 365, Notion |
| **Discussion threading** | Structured asynchronous dialogue with replies and reactions | LMS forums, Discourse, Slack channels |
| **Peer review workflow** | Submit → assign reviewers → review with rubric → revision | Turnitin, LMS peer review, Peergrade |
| **Collaborative whiteboard** | Visual co-creation: concept maps, affinity diagrams, process maps | Miro, Mural, FigJam |
| **Wiki / knowledge base** | Collectively built reference resource | Notion, Confluence, MediaWiki |
| **Collaborative annotation** | Group reading with shared comments anchored to text | Hypothes.is, Perusall |
| **Group project space** | Shared workspace with tasks, files, communication | Teams, Slack + drive, Basecamp |
| **Live polling / co-sensemaking** | Real-time input aggregated and displayed | Mentimeter, Slido, Poll Everywhere |

## Design Principles

- **Structure the interaction** — "collaborate in the shared doc" is not a design. Specify: who does what, by when, using what criteria, producing what artifact.
- **Individual accountability within group work** — design so that each person's contribution is visible. This prevents free-riding and enables assessment.
- **Roles rotate** — assign roles (facilitator, note-taker, devil's advocate, synthesizer) and rotate them across activities.
- **Asynchronous ≠ absent facilitation** — async collaboration needs [teaching presence](../../principles/community-of-inquiry.md): the instructor summarizes, redirects, asks deeper questions, and connects threads.
- **Shared artifacts > discussion for discussion's sake** — groups that build something together (a document, a framework, a recommendation) produce deeper learning than groups that only discuss.
- **Accessibility** — all collaborative tools must be keyboard-navigable and screen-reader compatible. Real-time collaboration tools often have accessibility gaps — test and provide alternatives.

## Group Decision-Making Interactions

When the collaborative task involves a group DECISION (not just discussion):
- **Structured deliberation**: present options → individual evaluation → group discussion → vote or consensus
- **Devil's advocate role**: assign someone to argue against the emerging consensus to prevent groupthink
- **Anonymous input → revealed discussion**: collect individual positions anonymously first, then discuss — prevents anchoring on the first speaker
- **Decision visualization**: use shared [decision matrices](../visual/decision-visualization.md) or trade-off charts to make the group's reasoning visible
- **Debrief the decision process**: after the decision, review HOW the group decided, not just WHAT they decided

## Common Mistakes

- **Unstructured groups** — "Work together on this" without roles, criteria, or deadlines
- **Tool overload** — using 5 different collaboration tools in one course; each requires its own cognitive overhead
- **Grading the group, not the individual** — unfair and reduces accountability
- **No facilitation in async** — the discussion board becomes a ghost town or a series of disconnected posts
